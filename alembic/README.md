# Alembic migrations — NESpresso bot

Schema migrations for the Postgres database (SQLAlchemy 2.0 async + AsyncPG).

## What's wired up

- `alembic.ini` (repo root) — standard config. The `sqlalchemy.url` in it is a
  **placeholder**; `env.py` overrides it at runtime from
  `nespresso.core.configs.settings.settings.POSTGRES_DSN` (the same DSN the app
  uses), so credentials stay in `.env` only.
- `env.py` — sets `target_metadata = Base.metadata` and imports
  `nespresso.db.models` so every model (including `profile_reaction`) is
  registered for `--autogenerate`. Uses the async engine template
  (`async_engine_from_config` + `asyncio.run` + `run_sync`), keeping the app's
  own `postgresql+asyncpg` DSN. Both offline (`--sql`) and online modes work.
- `versions/` — migration scripts land here (currently empty except `.gitkeep`).

Run from the **repo root**. In the Docker image `PYTHONPATH=/usr/src/app/src`
already makes `nespresso` importable; `env.py` also prepends `src/` to
`sys.path` so local runs work without setting `PYTHONPATH`. A valid `.env` (or
the equivalent env vars) must be present because importing settings loads it.

## Relationship to `EnsureDB()` — read this first

`db/session.py::EnsureDB()` (a `create_all` + hand-written `ALTER TABLE
IF NOT EXISTS` / index migrations, run at every startup) **stays as-is for
now.** Alembic is *additive*: it is being introduced alongside `EnsureDB()`, not
as a replacement, until migration parity with the current schema is proven and
verified against the live DB. Do **not** remove `EnsureDB()` as part of adding
this scaffold. Until the cutover, treat `EnsureDB()` as the source of truth for
the schema and Alembic as the mechanism for *new* changes going forward.

## One-time baseline (run against the live DB, post-deploy)

The live DB already has all tables (created by `EnsureDB()`), so we must **not**
let the first migration try to re-create them. Baseline it instead:

1. **Generate the baseline migration** (autogenerate diffs `Base.metadata`
   against the live schema — needs a DB connection):

   ```bash
   alembic revision --autogenerate -m "baseline"
   ```

2. **Review the generated script in `versions/` carefully.** Because
   `EnsureDB()` and the ORM models should already agree, ideally the `upgrade()`
   is (near) empty. Anything it wants to create/drop is drift between the models
   and what `EnsureDB()` built — reconcile it, but do **not** blindly apply
   table drops/creates against the populated production DB.

3. **Stamp the DB at that revision without running it** (marks the existing
   schema as already at `head`, so Alembic won't try to build tables that exist):

   ```bash
   alembic stamp head
   ```

4. From here on, author every schema change as a migration:

   ```bash
   alembic revision --autogenerate -m "describe change"   # review the diff!
   alembic upgrade head                                    # apply
   ```

   Always eyeball the autogenerated script — Alembic doesn't detect everything
   (e.g. some server-default / type changes, partial/`WHERE` indexes like
   `ix_tg_user_nes_id_verified_uniq`). Add those by hand where needed.

### Offline / SQL-only

To emit SQL without connecting (e.g. for DBA review):

```bash
alembic upgrade head --sql > migration.sql
```

## Sanity checks

```bash
alembic current      # show the DB's current revision (needs DB)
alembic history      # list revisions
alembic heads        # confirm a single head
```
